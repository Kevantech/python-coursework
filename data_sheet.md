# Datasheet: Black-Box Optimisation Query Data

This datasheet describes the dataset used in my black-box optimisation (BBO) capstone project. Unlike a conventional dataset collected from the real world, this dataset was generated interactively by me: it consists of the input points I chose to query and the outputs returned by a set of hidden functions.

## Motivation

- **For what purpose was the dataset created?** The dataset was created as part of the capstone project of my machine learning programme. Its sole purpose is to support the optimisation of a set of unknown black-box functions: each data point is one query (an input vector) and the function value returned for it. The data allows surrogate models to be trained so that future queries can be chosen intelligently.
- **Who created the dataset and on behalf of which entity? Who funded it?** The input points were selected and submitted by me (the student) through the capstone project portal. The corresponding outputs were generated by the hidden evaluation functions provided by the programme organisers, who designed and host the black-box challenge. No external funding is involved.

## Composition

- **What do the instances represent?** Each instance is a single query: a vector of continuous input values within a bounded domain, paired with the single numerical output the black-box function returned for that input. There is one such table per black-box function.
- **How many instances of each type are there?** The challenge comprises multiple functions of increasing input dimensionality (from 2 dimensions up to 8). For each function, the number of instances equals the number of queries submitted over the course of the programme — a small dataset by design, since the query budget is deliberately limited. *[Insert your exact per-function query counts here, e.g. "approximately 20–30 queries per function".]*
- **Is there any missing data?** No. Every submitted query received a valid numerical response, so the tables are complete. (If a submission round were ever missed, the corresponding rows would simply not exist rather than contain missing values.)
- **Does the dataset contain confidential data?** No. The data contains no personal information, no communications and nothing about individuals — only abstract numerical inputs and outputs of synthetic functions.

## Collection process

- **How was the data acquired?** Interactively and iteratively. In each round, I selected candidate input points (initially by space-filling random/Sobol sampling, later guided by a Bayesian optimisation surrogate model), submitted them via the capstone portal, and recorded the returned outputs in this repository.
- **If the data is a sample of a larger subset, what was the sampling strategy?** The data is a very small, deliberately biased sample of the functions' full input domains. Early queries used approximately uniform space-filling sampling to cover the domain; later queries were selected by an acquisition function (Expected Improvement / Upper Confidence Bound) that concentrates sampling in regions predicted to have high values or high uncertainty. This adaptive strategy is intentional — it is the whole point of the project — but it means the sample is not representative of the domain as a whole.
- **Over what time frame was the data collected?** Across the Stage 2 submission rounds of the programme (Modules 12–24), with one batch of queries submitted per round. *[Insert your actual date range here.]*

## Preprocessing/cleaning/labelling

- **Was any preprocessing/cleaning/labelling done?** Minimal. Inputs were kept in their native scale for record-keeping; for modelling, inputs were normalised to the unit hypercube and outputs were standardised (zero mean, unit variance) before fitting the Gaussian process surrogate, as GPs are sensitive to scale. No instances were removed and no labels beyond the returned function values exist.
- **Was the raw data saved in addition to the preprocessed data?** Yes. The raw query logs (exact submitted inputs and returned outputs, per round) are stored as CSV files in the `data/` folder of this repository; all normalisation is done on the fly in the modelling code, so the raw record is always preserved.

## Uses

- **What other tasks could the dataset be used for?** The query logs could be reused to benchmark alternative optimisation strategies offline (e.g. comparing how a different surrogate model or acquisition function would have chosen points), to study exploration–exploitation behaviour, or as a teaching example of sequential experimental design.
- **Is there anything about the composition or collection that might impact future uses?** Yes: the dataset is small and adaptively sampled, so it heavily over-represents high-value regions of each function and under-represents the rest of the domain. Any model trained on it will interpolate poorly in unexplored regions. Because the data describes synthetic functions rather than people, there is no risk of unfair treatment of individuals or groups.
- **Are there tasks for which the dataset should not be used?** It should not be used to draw conclusions about the global shape of the hidden functions outside the sampled regions, and it has no validity for any real-world prediction task — the functions are synthetic challenge objectives, not measurements of any real phenomenon.
